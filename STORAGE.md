# Хранилище DocSets

Документ определяет формат хранилища DocSets следующего поколения. Текущий
формат `*.docsets.json` считается устаревшим и не читается новым основным
приложением. Преобразование выполняет отдельная утилита `DocSets.Import.exe`.

## Цели

- DocSet самодостаточен и переносится копированием одного каталога.
- Пути по возможности остаются относительными.
- Одновременно можно открыть несколько DocSet и несколько источников кода.
- Пользовательские настройки интерфейса не смешиваются с переносимыми данными.
- Содержимое заметок и двоичные ресурсы могут развиваться независимо от дерева.

## Структура каталога

```text
Vital.DocSets/
|-- docsets.json
|-- content/
|-- assets/
`-- attachments/
```

`docsets.json` — обязательный манифест. Остальные каталоги создаются по мере
необходимости. Чтобы перенести документ целиком, пользователь копирует весь
каталог `Vital.DocSets`.

Упаковка в ZIP пока намеренно не рассматривается.

## Манифест

Манифест содержит явный маркер и версию формата:

```json
{
  "format": "DocSets",
  "formatVersion": 1,
  "id": "vital",
  "name": "Vital",
  "sources": [],
  "tags": [],
  "items": []
}
```

Новое хранилище принимает только явно поддерживаемые формат и версии. Оно не
пытается автоматически интерпретировать устаревшие файлы.

## Элементы и содержимое

Дерево хранится плоским списком элементов со стабильными читаемыми ID и
ссылками на родительские ID. Пользовательский текст заметки называется
`content`, а не `comment`.

```json
{
  "id": "user-service-save",
  "parent": "services",
  "name": "UserService.Save",
  "contentFormat": "Markdown",
  "content": "Вызывает внешний сервис идентификации."
}
```

Версия 1 поддерживает встроенный Markdown. Поле `contentPath` зарезервировано
для форматов, содержимое которых хранится в отдельных файлах. Один элемент не
может одновременно содержать непустые `content` и `contentPath`.

## Источники кода

DocSet может ссылаться на несколько источников кода:

```json
{
  "id": "backend",
  "name": "Backend",
  "type": "Solution",
  "root": "../Backend",
  "path": "Backend.sln"
}
```

`root` задаётся относительно каталога DocSet. Абсолютная локальная подмена
может быть представлена настройкой рабочего пространства. `path` задаётся
относительно `root`. Пути закладок также задаются относительно корня выбранного
источника. При перемещении всего дерева исходников обычно достаточно изменить
только `root`.

Пустой или отсутствующий `root` равнозначен `"."` и обозначает сам каталог
`Test.DocSets`. Значение `".."` обозначает каталог, в котором расположен
`Test.DocSets`.

Элементы ссылаются на источники по стабильному ID:

```json
{
  "sourceId": "backend",
  "project": "Backend.Core",
  "path": "Core/Services/UserService.cs",
  "symbol": "Backend.Core.Services.UserService.Save"
}
```

`sourceId` определяет корень solution, проекта или каталога. Необязательное
поле `project` по-прежнему определяет проект внутри solution и используется для
устранения неоднозначности символов.

### Основной источник

У каждого DocSet один источник является основным (`source-default`). Для новых
документов рекомендуется использовать стабильный ID `default`. Если у старого
документа только один Source, он считается основным независимо от его ID и без
перезаписи манифеста.

Пустой `sourceId` ссылки всегда означает основной источник текущего DocSet.
Для основного источника `sourceId` не записывается. Явный `sourceId` требуется
только для дополнительных источников. Если Sources несколько, но источник
`default` отсутствует, пустая ссылка считается неоднозначной.

При создании ссылки выбирается наиболее конкретный Source, содержащий файл.
Если выбран основной Source, ссылка остаётся короткой. Для дополнительного
источника используется форма `file:<sourceId>|<relative-path>`.

### Копирование между DocSet

JSON-буфер DocSets содержит служебный снимок корней Sources исходного документа.
При вставке исходная ссылка временно разрешается до абсолютного пути в памяти,
после чего пересчитывается относительно Sources целевого DocSet:

- ссылка на основной источник сохраняется без `sourceId`;
- ссылка на дополнительный источник получает его ID;
- если подходящего Source нет, ссылка становится явной внешней абсолютной ссылкой.

Абсолютный промежуточный путь не добавляется в обычный Markdown или текстовый
буфер. Обычное копирование остаётся читаемым и сохраняет относительные ссылки.

Таблица преобразования путей для частичного перемещения дерева откладывается до
появления подтверждённой необходимости. В версии 1 у каждого источника один
корень.

## Рабочее пространство

Рабочее пространство — локальная сессия, а не DocSet. Оно содержит список явно
открытых DocSet, активный DocSet, временные подмены источников, выделение и
раскладку интерфейса.

Visual Studio хранит локальное рабочее пространство в:

```text
<solution>/.vs/DocSets/workspace.json
```

Файл имеет собственные маркер и версию:

```json
{
  "format": "DocSetsWorkspace",
  "formatVersion": 1,
  "openDocSets": [
    "docs/Vital.DocSets",
    "D:/Shared/Shared.DocSets"
  ],
  "activeDocSet": "docs/Vital.DocSets",
  "sourceRootOverrides": {},
  "ui": {}
}
```

Относительные пути считаются от каталога solution, а не от вложенного каталога
`.vs/DocSets`. DocSet внутри репозитория сохраняется относительным путём. DocSet
за пределами репозитория сохраняется абсолютным путём.

Desktop-приложение хранит последние локальные сессии в `%LocalAppData%`.
Переносимый `*.docsets-workspace.json` может появиться позднее, но не требуется
версией формата 1.

DocSet открываются и создаются явно. Поиск может предложить найденные рядом
DocSet, но не подключает их и не выбирает место записи автоматически.

## Ресурсы

Ресурсы принадлежат одному DocSet и находятся в `assets/`. Имена файлов
формируются из хеша содержимого, поэтому одинаковые данные хранятся внутри
одного DocSet только один раз.

```text
assets/images/31a40fc2...png
```

Редакторы используют логические ссылки:

```markdown
![Диаграмма](asset:images/31a40fc2...png)
```

При копировании элемента специальный формат буфера DocSets включает данные всех
используемых ресурсов. Двоичные данные находятся только в специальном JSON-
буфере. Обычный текстовый и Markdown-буфер содержит короткую логическую ссылку,
но не Base64 и не абсолютный путь. При вставке в другой DocSet недостающие
ресурсы импортируются в целевое хранилище. Один путь в файловой системе не
считается полноценным буфером обмена между DocSet.

Версия 1 принимает PNG, JPEG, GIF и WebP. Изображение из системного буфера
сохраняется как PNG, предоставленный браузерным компонентом. Вставка одинаковых
данных повторно использует существующий файл по SHA-256.

Неиспользуемые ресурсы не удаляются автоматически. Будущая команда обслуживания
найдёт кандидатов и после подтверждения переместит их в локальный каталог
`.trash`.

## Гарантии сохранения

Манифест сначала записывается во временный файл внутри каталога DocSet, затем
повторно читается для проверки и только после этого атомарно заменяет предыдущий
манифест там, где платформа поддерживает такую операцию. Неудачная запись не
должна повреждать предыдущий читаемый манифест.

## Импорт устаревшего формата

Поддержка старого формата изолирована в отдельном исполняемом файле:

```text
DocSets.Import.exe old.docsets.json --output Vital.DocSets
```

Все старые DTO и правила совместимости принадлежат импортёру. Основное
приложение и новое хранилище не получают устаревших псевдонимов JSON и ветвей,
предназначенных только для миграции.
